## 详细功能说明

下面按页面和业务能力对当前系统已经实现的功能做完整说明。

### 1. 登录、账号与权限控制

- 支持账号密码登录和基于 JWT 的接口鉴权。
- 后端实现接口级权限校验、TOE 级访问控制。
- 系统区分 `admin` 与普通用户两类角色。
- 管理员可以创建用户、重置或修改密码、删除用户。
- 管理员可以按 TOE 维度为普通用户分配 `read` 或 `write` 权限；管理员默认拥有全部 TOE 的读写权限。
- 首次登录后的密码修改约束。

### 2. TOE 管理

- 支持创建、编辑、删除 TOE，并按名称、类型进行查看与筛选。
- TOE 基础信息支持维护版本、产品简介、TOE 类型说明等字段。
- TOE 详情页提供总览、资产、TOE 文档、ST/PP 文档等多个标签页，便于把结构化信息和证据资料放在同一个上下文中维护。
- 支持 `AI Consolidate` 对整段描述进行收敛优化（主要用于ST/PP内容提取后的内容整合。），也支持按字段逐项 AI 补写。
- 支持将中文字段整体翻译为英文，适合做英文版 ST 草稿准备。
- 支持导出单个 TOE 的打包文件并再次导入，便于跨环境迁移、备份或做演示数据分发。

### 3. 资产与资料管理

- 支持为 TOE 建立资产清单，并记录资产名称、类型、重要度等信息。
- 资产可以作为 Threat 识别和完整性分析的直接输入。
- 支持上传产品手册、设计说明、参考资料、ST/PP 文档等文件。
- 文件上传后支持处理状态跟踪，便于区分已解析、解析中、失败等状态。
- 上传文件统一落盘到 `storage/` 目录，并带有扩展名白名单、文件名清洗和路径安全处理。
- 可以对 TOE 文档与 ST/PP 文档分别管理，使“产品资料”和“参考基线资料”两个语义分开。

### 4. Threat / Assumption / OSP 与 SPD 管理

- Threat 页面分为 `Threats`、`Assumptions`、`OSPs`、`Completeness` 四个核心区域。
- Threat 支持录入威胁编号、威胁主体、负面动作、受影响资产、后果与风险等级。
- Threat 支持确认、回退、标记误报、批量确认、批量删除以及风险等级人工覆写。
- Assumption 与 OSP 支持独立新增、编辑、删除、确认、驳回、回退和批量确认/驳回。
- Threat、Assumption、OSP 都可以映射到后续的 Security Objective，保证安全问题与目标之间有来源链路。
- Threat 完整性检查会从资产覆盖、目标映射、SFR 映射、测试覆盖、残余风险与参考文档对齐多个角度给出评分和提示。
- 对弱覆盖资产支持忽略开关，避免某些已知不适用项持续干扰完整性评估。

### 5. ST 参考资料与文档导入能力

- 支持上传已有 ST 参考文档并提取 Threat、Objectives、SFR、Assets 等结构化内容。
- 参考资料支持解析重试、删除和共享开关，便于团队复用。
- Threat 支持从文档导入，Security Objectives 也支持从 ST/PP 文档导入。
- 这套机制适合把历史项目经验转成当前项目的候选项，而不是每次都从零开始。

### 6. 安全目标与 SFR 管理

- Security 页面分为 `Objectives`、`SFRs`、`Completeness`、`SFR Library` 四个核心区域。
- Security Objective 支持新增、编辑、删除、确认、驳回、回退。
- 每个 Objective 都可以查看其来源映射，明确它是由 Threat、Assumption 还是 OSP 推导出来的。
- 支持手工映射 Objective 来源，也支持解除映射，方便审查过程中做精修。
- 支持 AI 推荐安全目标，也支持从 ST/PP 文档导入目标。
- SFR 实例支持创建、编辑、删除、确认、驳回、回退，并可映射到一个或多个 Objective。
- 支持按 Objective 自动匹配 SFR、AI 补全 SFR 内容、自动补齐依赖、ST/PP 一致性校验。
- 导出时会自动利用 SFR 依赖链生成更完整的结果内容。

### 7. SFR 库管理

- 系统内置独立的 SFR Library 页面，不只是查看标准条目，还支持维护与治理。
- 支持分页检索、编辑、删除、批量删除。
- 支持通过 CSV 文件导入 SFR 条目，便于批量构建或修正企业内部库。
- 依赖字段会在后端统一规范化，只保留标准化的 SFR ID，并保留 `AND/OR` 逻辑。
- 这个库既可以当标准参考库，也可以当团队裁剪后的落地知识库使用。

### 8. 测试用例管理

- Tests 页面分为测试列表和完整性检查两个区域。
- 测试用例支持新增、编辑、删除，并维护测试编号、测试名称、测试目标、测试对象、测试场景、前置条件、步骤、预期结果等字段。
- 支持把一个测试用例关联到多个 SFR。
- 测试类型支持 `coverage`、`depth`、`independent` 三类，可组合使用。
- 支持单条确认、驳回、回退，也支持批量确认和批量删除。
- 支持 AI 按当前 SFR 集合批量生成测试用例，并在页面中展示任务进度。
- 测试完整性报告会分析当前测试对 SFR 的覆盖情况，帮助定位缺口。

### 9. 风险看板与保障链分析

- Risk 页面不是简单统计页，而是对当前 TOE 的建模质量做集中体检。
- 看板会展示 Assurance Maturity 指标，把 Threat 完整性、Security 完整性、Test 完整性等维度聚合成整体分值。
- 提供 Assurance Chain 可视化，用链式方式展示从安全问题到目标、SFR、测试的衔接强度。
- 提供 Assurance Tree，便于从层级视角理解哪一层最薄弱。
- 提供 Blind Spots 检测，识别资产、Threat、Objective、SFR、Test 等维度可能遗漏的点，并给出置信度。
- 提供风险分布、残余风险状态、完整性雷达图、关键发现、AI 总结等分析内容。
- 支持记录 Threat 对应的残余风险处理结论，例如接受、缓解、转移或规避。

### 10. AI 模型与 AI 辅助能力

- 每个用户都可以配置自己的 AI 模型，不强依赖单一厂商。
- 只要兼容 OpenAI 协议，就可以接入 OpenAI、Ollama、vLLM 或其他自托管服务。
- 支持新增、编辑、删除、连通性校验、设置当前工作模型。
- API Key 在后端按加密方式保存，不以明文落库。
- 提供模型级聊天窗口，便于在接入后先做快速验证。
- 当前已经落地到业务流程中的 AI 能力包括：
  - TOE 描述草稿生成与字段收敛
  - 资产建议
  - 文档分析
  - Threat 扫描
  - Assumption / OSP 建议
  - Security Objective 建议
  - SFR 匹配与补全
  - Test Case 生成
  - 英文翻译
  - 风险总结与盲点建议
- 所有耗时 AI 操作均通过任务状态轮询反馈进度，避免页面长时间卡死。

### 11. 导出、模板与交付能力

- Export 页面支持先预览、再编辑、最后导出。
- 支持导出 Markdown 版本和 Word `.docx` 版本的 ST 文档。
- 支持在导出前直接修改 Markdown 内容，适合在系统生成结果基础上做最后润色。
- ST 模板可在设置页集中维护，系统已经支持一组占位符来映射 TOE、资产、Threat、目标、SFR、测试等内容。
- 可以从文件加载模板，也可以把当前模板保存到文件，便于团队共享模板。

### 12. 设置、日志与系统运维能力

- 设置页目前整合了语言切换、AI 模型、用户管理、系统参数、日志查看、ST 模板维护等能力。
- 语言切换已支持中英文界面，适合国内建模与英文交付并存的场景。
- 管理员可以配置 PDF 解析超时时间，减少大文档处理时的失败概率。
- 系统提供审计日志和错误日志查询与清理能力，便于定位谁在什么时间改了什么内容，以及服务运行期间发生了哪些异常。
- 审计日志会记录用户、资源、IP、状态码、耗时等关键信息。

### 13. 安全性与工程化细节

- 后端对上传路径、文件名、扩展名进行了基础安全控制，降低目录穿越和恶意文件风险。
- 接口统一采用结构化响应格式，便于前端稳定处理。
- 生产环境下支持通过 Docker Compose 启动数据库、Redis、后端、Worker 与前端服务。
- `storage/`、`ST Template.md` 等数据通过卷挂载保留，方便持久化。
- 项目附带后端测试用例，覆盖认证、配置、加密、健康检查、日志、系统设置、TOE 与上传安全等关键点。
