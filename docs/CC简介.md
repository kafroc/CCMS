## CC(Common Criteria) 简介

Common Criteria（简称 CC，正式名称为 ISO/IEC 15408）是信息技术安全评估的国际标准。它被形象地称为信息安全界的“通用语言”或“度量衡”，旨在为IT产品的安全功能和保障措施提供一套统一、客观、可量化的评估框架。
Common Criteria (ISO/IEC 15408) 是全球信息安全领域互认程度最高的标准，也是全球政企、军事及金融领域公认的最权威安全评估准则。
- **官网**：[www.commoncriteriaportal.org](https://www.commoncriteriaportal.org/)
- **国际互认 (CCRA)**：目前全球有 31 个国家签署了互认协议，实现“一次评估，全球通行”。
- **巨头背书**：
  - **Apple**：其 iOS、macOS 以及 M 系列芯片中的 **Secure Enclave** 长期维持 CC 认证。
  - **Google**：**Pixel** 手机搭载的 **Titan M2** 安全芯片通过了 CC PP0084 认证。
  - **其他**：包括 **Microsoft Windows 11**、**Samsung Knox** 及 **Cisco** 核心网络设备，均视 CC 认证为产品进入高端市场的必备资质。

### 核心术语 (Glossary)
| 术语 | 全称 | 定义 |
| :--- | :--- | :--- |
| **TOE** | Target of Evaluation | **评估对象**。即接受评估、作为安全审计主体的 IT 产品或系统。 |
| **ST** | Security Target | **安全目标文档**。由开发者编写，定义了特定 TOE 的安全要求、保障措施以及与环境的边界。 |
| **PP** | Protection Profile | **保护轮廓**。由行业组织或政府发布的通用安全要求模板，规定了某类产品（如防火墙）必须满足的标杆。 |
| **SPD** | Security Problem Definition | **安全问题定义**。对 TOE 及其运行环境面临的威胁 (Threats)、组织安全策略 (OSPs) 和假设 (Assumptions) 的声明。 |
| **SFR** | Security Functional Requirements | **安全功能要求**。从 CC 标准第二部分选取的组件，定义了“安全功能需求模板”。 |
| **SAR** | Security Assurance Requirements | **安全保障要求**。从 CC 标准第三部分选取的组件，规定产品“如何被开发和验证”（如：代码审计、渗透测试）。 |
| **EAL** | Evaluation Assurance Level | **评估保障等级**。从 EAL1 到 EAL7，代表对产品安全特性的评估严谨程度和置信度。 |
| **Operational Environment** | - | **运行环境**。TOE 运行所需的外部环境，通常通过 OE Objective（环境安全目标）来规范。 |

### CC的安全哲学
当我们在向客户销售一款IT产品，我们实际上很难向客户解释为什么这款产品是安全的，让客户可以放心使用。
CC标准用严谨的逻辑论证了怎么做产品是安全的，这样即使用户不懂安全技术，也能通过产品通过CC认证来得到安全信心。
![](./assets/Snipaste_2026-04-14_15-44-17.png)
所有的安全活动都是以资产为锚点，每个资产都是对应的所有者。威胁主体会对资产造成威胁，以致增加了资产面临的风险。所有者的目标是通过一些对策去消除/减小/转移风险。

![](./assets/Snipaste_2026-04-14_15-45-36.png)
资产所有者希望资产的风险是可接受的，也就是对资产产生足够安全的信心。而这个信心可以通过CC认证评估而得到增强。风险的降低需要一些处理控制对策，CC通过证明控制最小化风险的充分性，也就是说控制是充分的，所以风险能最小化。另外通过证明控制实现的正确性，也就是说控制是被正确实现的，所以风险做到了最小化。通过充分性和正确性评估后，理论上可以保障产品是安全的。

PS：资产的梳理，威胁的识别不像数学公式，还是有主观的空间，所以严格意义上讲并不是一款产品通过了CC认证就保证100%安全，没有漏洞，这种理解方式是不对的。CC更大程度上是一种形式化证明，逻辑上没有问题，但是安全是一个动态对抗的过程，攻击手段会增加，资产的范围会调整。CC的意义更重要的是把安全水位给显著提升，并不是要保证产品没有任何安全漏洞。

CC的安全逻辑链：
- TOE有资产 -> 这些资产会面临安全威胁 -> 为了解决这些威胁，需要制定安全目标 -> 为了实现安全目标，需要明确安全功能(SFR)。
- 那么如果安全功能被正确实现 -> 达成安全目标 -> 解决安全威胁 -> 保证资产安全
